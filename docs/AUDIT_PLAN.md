## 🔴 Критичні (безпека / витік даних)

### [x] 1. `pages/api/stats.js` — відкрите виконання довільного SQL без авторизації
POST на `/api/stats` бере `req.body.sql` і виконує його напряму через `execute_sql`
з `SUPABASE_SERVICE_KEY` — **без перевірки логіну, без ролі, без `validateReadOnlySql`**.
Будь-хто в інтернеті може прочитати всю БД (імена пацієнтів, діагнози).
**Виправлення:** додати перевірку авторизації (getUserRole), пропускати SQL через
`validateReadOnlySql`, для ролі doctor — застосовувати той самий фільтр що в ask.js,
або взагалі заборонити довільний SQL від клієнта (тільки whitelisted-запити).

### [x] 2. Роль `doctor` бачить дані всієї лікарні через аналітичні VIEW
Фільтр по `doc_name` застосовується тільки якщо в SQL є `FROM lsmd`. Але роутер
для більшості запитів повертає готові VIEW (`v_department_stats`, `department_diagnoses`,
`v_diagnosis_stats`, `v_hospital_summary`), які `lsmd` не містять → фільтр не спрацьовує
→ doctor (Ільніцька) бачить статистику всіх відділень і лікарів.
**Виправлення:** для ролі doctor дозволяти тільки запити з власною фільтрацією по lsmd
(або власні doctor-VIEW), блокувати агрегати по всій лікарні. Розглянути RLS на рівні БД.

### [x] 3. Фронтенд `index.js` не знає про роль
Усі 80+ прикладів (зокрема «Скільки пацієнтів пролікував Грабовський») показуються будь-кому.
`provider = 'groq'` захардкоджено, роль ніде не зчитується.
**Виправлення:** зчитувати роль через app_users, фільтрувати ALL_EXAMPLES по ролі,
ховати приклади «по інших лікарях» для doctor.

## Поглиблений захист БД (RLS-альтернатива, варіант B)

### [x] 13. service_role обходив застосунковий захист + execute_sql була публічною
**Виявлено:** `execute_sql` мала EXECUTE для anon/PUBLIC → будь-хто з браузерним
anon-ключем міг виконати довільний SELECT напряму через Supabase REST, оминаючи
весь застосунковий захист. Плюс простий RLS не спрацював би, бо бекенд ходить
через service_role (обходить RLS за дизайном).

**Зроблено (міграція execute_sql_safe_with_role_guard):**
- Створено `execute_sql_safe(sql_query, p_role, p_doc_name)` — перевірка ролі
  ВСЕРЕДИНІ БД (другий рубіж). Для doctor: запит мусить читати lsmd І містити
  фільтр по власному doc_name, інакше RAISE EXCEPTION. Дублює guard на SELECT-only
  та single-query на рівні БД.
- Відкликано EXECUTE на старій `execute_sql` від PUBLIC/anon/authenticated
  (лишились service_role + postgres).
- Нова функція доступна тільки service_role.
- ask.js / stats.js / icu.js переключено на execute_sql_safe з передачею ролі.

**Перевірено (6 тестів):** admin бачить усе ✓; doctor бачить свої дані ✓;
doctor→загальна VIEW відмова ✓; doctor→чужий лікар відмова ✓;
viewer→загальна пройшов ✓; multi-query "; DROP" відмова ✓.

### [x] 14. Баги, виявлені на проді (скріншот користувача)
**Проблема 1 — "скільки я пролікувала інсультів" → хибна відмова:**
regex `\b(я|мої|...)\b` НЕ працює з кирилицею в JS (\b не визначає межу для не-ASCII),
тому підстановка прізвища для doctor не спрацьовувала → запит ішов у діагнозну VIEW
(відмова) або в LLM. Виправлено: кириличе-сумісні межі `(^|[^а-яіїєґ'])...([^а-яіїєґ']|$)`.

**Проблема 2 — "летальність моїх пацієнтів" → Bad escaped character in JSON:**
коли роутер повертав NULL, запит ішов у LLM, який генерував SQL з невалідним
JSON-екрануванням (\'). JSON.parse падав. Виправлено: санітизація \' та інших
невалідних escape перед парсингом + кількаступеневий fallback.

**Проблема 3 (виявлено при тестуванні) — БД-функція відхиляла ВСІ doctor-запити:**
execute_sql_safe вимагала повного імені ('Ільніцька О. В.'), а застосунок фільтрує
по скороченому прізвищу ('%ільніцьк%') від detectDoctor → 100% відмов на проді.
Виправлено (міграція execute_sql_safe_surname_match_fix): перевірка по стему прізвища.

**Також:** прибрано дублювання doctor-фільтра в routeQueryWithRole.

**Перевірено (5 сценаріїв БД):** doctor скороч.прізвище/повне імʼя — проходять;
чужий лікар/загальна VIEW — відмова; admin — проходить.

### [x] 16. Нова модель доступу + достовірність + ПІБ власних пацієнтів
**Вимоги:** без помилок; достовірні відповіді; лікар бачить власних пацієнтів + загальні агрегати.

**Модель доступу doctor (БД execute_sql_safe + ask.js, два рубежі):**

**ПІБ власних пацієнтів:** додано patient_name у список випадків по діагнозу

**Доступ до агрегатів повернено:** /api/stats та /api/icu більше не блокують doctor

**Достовірність — прогнано всі 39 унікальних SQL роутера через БД:**

**Тести ізоляції (БД):** doctor→агрегат OK; doctor→свій lsmd OK; doctor→lsmd без

### [x] 17. Фінальна перевірка сценаріїв лікаря + закрито UNION/підзапит-обхід
Відтворено всі 18 сценаріїв лікаря через задеплоєну логіку роутера + execute_sql_safe:
- статистика (пролікував/нічні/вихідні/летальність/операції) — OK;
- списки власних пацієнтів з ПІБ (інсульт/епілепсія) — OK;
- спеціалізація — OK;
- загальні агрегати (топ діагнозів/летальність відділень/реанімація) — OK (дозволені);
- спроба чужого лікаря (Грабовський) — BLOCK.

**Виявлено й закрито 2 вектори обходу (критично):**
- UNION-обхід: `WHERE doc_name ILIKE '%свій%' UNION SELECT patient_name FROM lsmd`
  пропускався (фільтр у 1-й частині, 2-га читала всіх). Заборонено UNION/EXCEPT/INTERSECT.
- Підзапит-обхід: множинне звернення до lsmd. Заборонено >1 входження lsmd для лікаря.

**Фінальні тести ізоляції — усі 7 правильні:** L1 всі пацієнти BLOCK, L4 JOIN BLOCK,
L5 підзапит BLOCK, L6 UNION BLOCK; свій список/статистика/агрегат — ПРОЙШОВ.
Заборони продубльовано в ask.js (зрозумілі повідомлення замість БД-помилок).

### [x] 19. Метрика "нічні поступлення" недостовірна (дані)
admission_ts відображає час реєстрації в системі (пік 9-12 год), не фактичного
надходження. "Нічних" виходить 15/625 (2.4%) — формально вірно за SQL, але вводить
в оману. Потребує рішення: прибрати метрику для лікаря / перейменувати на "за часом
реєстрації" / лишити. ОЧІКУЄ відповіді користувача.

**Рішення #19 (виконано):** реалізовано логіку чергувань на запит користувача:
- доба чергування = 09:00→09:00 (надходження до 09:00 належить попередній добі);
- нічні поступлення = 22:00–07:00 (було 22:00–06:00);
- гілка "нічні": показує нічних_поступлень + діб_чергування_з_нічними;
- нова гілка "доби чергування": COUNT DISTINCT діб 09:00→09:00 (Ільніцька 2024: 204 доби, 625 поступлень; 15 нічних на 14 діб);
- назва чесно позначена "за часом у системі" (admission_ts = час реєстрації, пік 9-12).

### [x] 20. Приклади для лікаря: прибрано непрофільні (хірургія/урологія/тощо)
Лікар-невролог бачив у прикладах хірургію, апендицити, простату, урологію — бо для

прибрано "операції у мене" (невролог не оперує).
Виправлено баг розпізнавання: "транзиторна ішемічна атака" ловилась як ішемічний

### [x] 21. Групування показників по клінічних напрямках (терапія/хірургія)
ВИЯВЛЕНО: класифікація вже повністю існує в БД (з попередніх сесій):

### [x] 22. Напрямки без року — за весь період (узгодження із загальною статистикою)

### [x] 24. Гілка "лікарі відділення"

### [x] 26. Номер відділення №1/№2 ігнорувався; хірургія плуталась

- гілка "відділення напрямку" тепер лише за явним словом напрям/блок (не перехоплює

### [x] 27. "Всіх хірургів" (спеціальність) vs "хірургія №1" (відділення)
"покажи всіх хірургів" давало статистику Хірургічного відділення №1 замість списку
всіх лікарів за фахом хірург. Виправлено розмежування спеціальність/відділення:
- deptMap пропускає спец-ключі (хірург/терапевт/уролог/травматол/гастро/гематолог/нейрохірург),
  якщо в запиті НЕМАЄ слова "відділ"/"центр"/номера → запит про фах, не відділення;
- гілка спеціальностей: точний матч specialization='хірург' (не ILIKE, щоб не захопити
  нейрохірурга), нейрохірург перевіряється раніше, додано тригер "покажи";
- результат за фахом сортується по відділеннях.
Перевірено: всі хірурги=65 (2 відділення, без нейрохірургів); нейрохірурги окремо;
"хірургія №1" → відділення; терапевти за фахом vs терапевтичне відділення №1 — різне.
