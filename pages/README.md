# Структура `pages/` — довідник

Карта всіх сторінок і API-маршрутів проєкту з поясненням, де що лежить і за що відповідає.

## Сторінки (UI)

### `_app.js`
Кореневий компонент Next.js. Підключає глобальні стилі (`styles/globals.css`) і обгортає всі сторінки. Сюди не лізти без потреби — тут немає логіки.

### `login.js`
Сторінка входу. Форма email/пароль → `supabase.auth.signInWithPassword`. При успіху редірект на `/`. Клієнтський Supabase через `lib/supabase.createClient()`.

### `index.js`
**Головний дашборд.** Найбільший файл (1151 рядків). Бічна панель (sidebar) з навігацією по 13 секціях аналітики, KPI-картки, графіки. Тягне дані через `POST /api/stats` з різними `key`. Містить вбудовані CSS-стилі дашборду (`DASH_CSS`) у дусі дизайн-системи ЛСМД. Показує профіль користувача (ім'я/роль) знизу сайдбару.

### `analytics.js`
Розширена аналітична сторінка з графіками на **Recharts** (BarChart, LineChart). Хук `useQuery(key)` тягне агреговані дані з `/api/stats`, `useInView` — лінива анімація появи блоків при скролі (IntersectionObserver). Іконки відділень — з `lib/dept-icons`.

### `glow.js`
Експериментальна "вітринна" візуалізація — анімована canvas-графіка (хвилі/частинки) з даними по лікарях/діагнозах. Хук `useStats(key)` — те саме, що `useQuery`, тягне `/api/stats`. Кольорові палітри `DOC_C`/`DIAG_C` для розфарбовування серій.

### `org.js`
Організаційна структура лікарні — картки відділень, згруповані по блоках (`хірургічний`, `терапевтичний`, `інтенсивна_терапія`, `параклінічний` — конфіг `BLOCK_CFG`/`BLOCK_ORDER`). Функція `initials(name)` робить ініціали з ПІБ для аватарок співробітників.

### `admit.js` ⭐ нове
**Форма госпіталізації пацієнта.** Двосекційна форма (дані пацієнта + параметри госпіталізації):
- `calcAge(birthDateStr)` — обчислює вік за датою народження у форматі "ДД.ММ.РРРР"
- `todayDMY()`/`nowHM()` — поточні дата/час у потрібному форматі за замовчуванням
- автопошук МКХ-10 з debounce 350мс через `/api/stats?key=icdSearch`, вибір коду через `pickIcd(row)`
- `submit(e)` — валідація (відділення обрано, ICD-10 обрано) і POST на `/api/admit`
- доступ лише для ролей `doctor`/`admin` (перевірка через `/api/me`)

### `cabinet.js` ⭐ нове
**Електронний кабінет лікаря.** Тягне дані з `/api/cabinet`:
- шапка з профілем лікаря (ПІБ, посада, спеціалізація, відділення)
- сітка статистики (`StatBox`): всього випадків, активних, серед. ліжкодень, ургентних, з поліпшенням, летальність
- перемикач вкладок: "Останні випадки" / "Активні пацієнти" / "Топ діагнози" — таблиці з кольоровими бейджами статусу (`StatusBadge`, мапа кольорів `STATUS_COLORS`)
- кнопка "+ Новий випадок" веде на `/admit`

## API-маршрути (`pages/api/`)

### `stats.js`
**Центральний API для читання аналітики.** Білий список SQL-запитів (захист від ін'єкцій):
- `QUERIES` — статичні запити (без параметрів)
- `PARAM_QUERIES[key](param)` — параметризовані (напр. `icdSearch` — пошук МКХ-10 за кодом/назвою для форми госпіталізації)
- `ADMIN_PARAM_QUERIES` — запити з ПІБ пацієнтів, доступні лише з роллю `admin` (`patSearch`, `patCard`, `patHistory`)

Усі параметри проходять через `esc()` (екранування лапок). Виконання — через RPC `execute_sql_safe`.

### `admit.js` ⭐ нове
**Створення нового випадку госпіталізації.** Логіка:
1. `getMe(req)` — визначає лікаря на сервері (ланцюжок `app_users → empl → lsmd_doctors`), клієнту довіряти не можна
2. `validate(b)` — перевіряє формати дат (`ДД.ММ.РРРР`), часу (`ГГ:ХХ`), коректність відділення/типу госпіталізації/коду МКХ-10
3. рахує наступний `id_case` (MAX+1, бо немає автоінкременту)
4. вставляє рядок напряму в `lsmd` через PostgREST REST (`SUPABASE_SERVICE_KEY`, в обхід RLS) — тригер `lsmd_sync_dates` сам обчислює похідні поля (дати, ліжкодень, зміна, тип дня)

Доступ — лише ролі `doctor`/`admin`.

### `cabinet.js` ⭐ нове
**Дані для кабінету лікаря.** `getMe(req)` визначає лікаря і його профіль (`full_name`, `position`, `specialization`, `department`). Білий список запитів `QUERIES`:
- `summary` — агрегована статистика (всього/активних/померло/поліпшення/середній ліжкодень/ургентних)
- `recent` — останні 12 випадків
- `active` — пацієнти, що зараз лікуються (до 25)
- `topDiag` — топ-6 діагнозів лікаря з таблиці `doctor_diagnoses` (JOIN по `doctor_id`)

Усі запити фільтруються по `doc_name = поточний лікар` — лікар бачить лише свої дані. `supaFetch(sql)` — обгортка над `execute_sql_safe`.

### `me.js`
Повертає дані поточного користувача: роль (`role`) і, якщо прив'язано до співробітника, — ПІБ/посаду/спеціалізацію з `empl`. Використовується майже на кожній сторінці для перевірки прав доступу і відображення профілю.

### `icu.js`
Агреговані показники реанімації (відділення `ILIKE '%анестез%'`) — без ПІБ пацієнтів, доступні всім авторизованим ролям. Для картки "Реанімація" на дашборді.

### `ask.js`
**AI-помічник із природномовними запитами.** Найбільший API-файл (587 рядків). Спочатку перевіряє роль користувача (`getUserRole`), потім намагається перехопити запит роутером (`lib/query-router.routeQueryWithRole` — ~11 категорій патернів, 0 токенів), якщо не вдалося — формує SQL через LLM (Groq/Gemini/OpenAI/Anthropic) із валідацією через `lib/sql-guard.validateReadOnlySql` (лише SELECT).

## Загальні принципи (для орієнтиру)

- **Авторизація**: завжди через `createServerClient` + cookies, роль визначається на сервері (`app_users.role`), ніколи не довіряти клієнту
- **Запис у БД**: лише через `pages/api/admit.js` напряму в PostgREST з service-role ключем (в обхід RLS), за білим списком полів і з валідацією
- **Читання даних**: лише через `stats.js`/`cabinet.js`/`icu.js` — параметризовані, екрановані запити з білого списку, ніколи "сирий" SQL з клієнта
- **Тема**: темний дизайн, CSS-змінні `--bg`, `--surface`, `--border`, `--accent`, `--text`, `--text3`, `--mono` (див. `styles/globals.css`)
